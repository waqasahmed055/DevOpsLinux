# Wiki Note: Map AD Users and Groups to POSIX Attributes on RHEL 8

This approach uses **POSIX attributes stored in Active Directory** and tells **SSSD on RHEL 8** to read those values instead of auto-mapping UIDs and GIDs. The linked guide describes this as the preferred method when you need consistent IDs across Linux systems, especially during migration from LDAP to AD direct authentication. ([OneUptime][1])

## Why we are doing this

If LDAP is being replaced by AD direct authentication, keeping the same numeric IDs is important so file ownership and access do not change. With this method, the user and group IDs come from AD attributes such as `uidNumber` and `gidNumber`, instead of being generated by SSSD mapping. ([OneUptime][1])

## AD attributes to set

For the user, the important attributes are:

* `uidNumber`
* `gidNumber`
* `unixHomeDirectory`
* `loginShell` ([OneUptime][1])

For the group, the key attribute is:

* `gidNumber` ([OneUptime][1])

## Example for UserA and GroupA

Use values that match your existing Linux UID/GID plan. The article shows PowerShell examples using `Set-ADUser` and `Set-ADGroup` to populate these fields in AD. ([OneUptime][1])

## RHEL 8 configuration

After adding the attributes in AD, configure SSSD on RHEL 8 to stop automatic ID mapping so it reads the AD POSIX values instead. The linked guide says to set `ldap_id_mapping = False` in the domain section of `sssd.conf`. ([OneUptime][1])

```ini
# /etc/sssd/sssd.conf

[domain/example.com]
id_provider = ad
ldap_id_mapping = False
fallback_homedir = /home/%u
ldap_user_shell = loginShell
default_shell = /bin/bash
```

Then clear cache and restart SSSD:

```bash
sudo sss_cache -E
sudo systemctl restart sssd
```

## Verify on RHEL 8

The article recommends checking the resolved UID/GID with `id` and confirming the account details with `getent passwd`. ([OneUptime][1])

```bash
id UserA
getent passwd UserA
getent group GroupA
```

## Notes

Use one method consistently across the environment: either SSSD auto-mapping or AD POSIX attributes. The linked guide recommends sticking to one approach to avoid UID/GID confusion and permission issues. ([OneUptime][1])
